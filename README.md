# Test Rails 8 Authentication (Minitest)

Companion sample app for the blog post [How to Test default Rails authentication with Minitest](https://minitestrails.com/blog/testing-rails-8-authentication-minitest/) at [minitestrails.com](https://minitestrails.com).

This is the Cookbook app from the [Minitest Rails guide](https://minitestrails.com/guide/introduction/), extended with Rails 8's built-in authentication generator, recipe ownership, and a full Minitest suite (integration + system tests).

## What this app demonstrates

| Layer | What is tested |
| --- | --- |
| **Authentication** | Sign-in, sign-out, bad password |
| **Protected routes** | Guests can browse. Signed-in users can create recipes |
| **Authorization (bonus)** | Only the recipe owner can edit or destroy |

No Devise, no Pundit — kept intentionally small so the focus stays on testing Rails built-in auth.

## Requirements

- Ruby **4.0.5** (see `.ruby-version`)
- Rails **8.1.3**
- Bundler
- **Chrome** (or Chromium) for system tests — Capybara drives headless Chrome via Selenium

Tested with Minitest 6.0.6 and selenium-webdriver 4.46.0.

## Quick start

```bash
git clone git@github.com:minitestrails/test-rails-default-auth-minitest.git
cd test-rails-default-auth-minitest
bin/setup --skip-server
bin/rails test:all
```

You should see **0 failures, 0 errors** when running the full test suite.

To run the app locally after setup:

```bash
bin/rails db:fixtures:load
bin/dev
```

Then open [http://localhost:3000](http://localhost:3000), you will land on the recipes index page with two recipes listed from the fixtures at `test/fixtures/recipes.yml`.

## Fixture users and passwords

All fixture users share the same plaintext password:

| Fixture | Email | Password |
| --- | --- | --- |
| `users(:alice)` | `alice@example.com` | `password` |
| `users(:bob)` | `bob@example.com` | `password` |

Recipes in fixtures:

| Fixture | Owner |
| --- | --- |
| `recipes(:pancakes)` | alice |
| `recipes(:lentil_soup)` | bob |

Use these in integration tests (`sign_in_as users(:alice)`) and system tests (`sign_in_to_ui_as users(:alice)`).

## Guest vs protected routes

| Action | Guest | Signed in |
| --- | --- | --- |
| `GET /recipes` (index) | Allowed | Allowed |
| `GET /recipes/:id` (show) | Allowed | Allowed |
| `new`, `create` | Redirect to sign-in | Allowed |
| `edit`, `update`, `destroy` | Redirect to sign-in | Owner only |

Implementation lives in `app/controllers/recipes_controller.rb`:

- `allow_unauthenticated_access only: %i[index show]`
- `ensure_recipe_owner` on `edit`, `update`, `destroy`

## Test helpers

### Integration tests — `sign_in_as`

Generated by the Rails 8 authentication generator. Wired in `test/test_helpers/session_test_helper.rb` and included automatically for integration tests.

```ruby
sign_in_as users(:alice)
get new_recipe_url
assert_response :success
```

Sets a signed `session_id` cookie without posting the sign-in form — fast and stable for protected-route tests.

### System tests — `sign_in_to_ui_as`

Defined in `test/application_system_test_case.rb`. Injects the same signed session cookie into the browser (after visiting a page so Selenium can set cookies).

```ruby
sign_in_to_ui_as users(:alice)
visit recipes_url
click_on "New recipe"
```

The **sign-in form itself** is tested once in `test/system/sessions_test.rb`. Protected-route system tests use `sign_in_to_ui_as` so the suite stays fast and does not re-test the login form on every case.

`teardown { Capybara.reset_sessions! }` clears browser cookies between tests so a signed-in session from one test does not leak into the next.

## Troubleshooting

**System tests fail with `Unable to find field ...`**

The sign-in form uses placeholder text, not labels. Match what `sessions/new.html.erb` renders:

```ruby
fill_in "Enter your email address", with: users(:alice).email_address
fill_in "Enter your password", with: "password"
```

**System tests pass alone but fail in the full file**

Usually session bleed between tests. Confirm `ApplicationSystemTestCase` has `teardown { Capybara.reset_sessions! }`.

**`Could not find Chrome` or Selenium errors**

Install Google Chrome or Chromium. System tests use `driven_by :selenium, using: :headless_chrome`.

**Database errors after switching branches**

Run the following command to reset the database:

```bash
bin/rails db:fixtures:load
```
